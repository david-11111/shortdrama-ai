# 下一步指引

写于 2026-06-20，系统核心链路和容错机制已通。

---

## 当前系统能力

```
用户创建项目
  → 选择 autopilot 模式
    → 自动：剧本 → 参考图 → 关键帧 → 视频 → 剪辑 → 成片
    → 遇到失败自动重试（3 次指数退避）
    → 重试失败自动换 provider（kling → seedance → joy-echo → ltx2.3 → wan2.1）
    → 全部失败自动跳过失败镜头，继续走完剩余
  → 用户拿到成片
```

**测试覆盖：** 621 单元测试 + 6 集成测试，全部通过。

---

## 可选的三条路

### 路 A：上生产（DevOps 纵深）

让系统真正能跑在线上不出事。

- Docker Compose 一键部署（PostgreSQL + Redis + FastAPI + Celery + Frontend）
- CI/CD pipeline（git push → auto test → deploy）
- Prometheus/Grafana 监控打点（返工次数、provider 失败率、run 耗时）
- 告警：run 卡住超过 N 分钟 → 通知
- 日志聚合（ELK / Loki / Grafana）
- 数据库迁移在 CI 中自动执行

**推荐理由：** 代码写好了不等于系统能在线上稳定跑。Docker Compose 是性价比最高的第一步——新环境 clone 下来 `docker compose up` 就能跑。

### 路 B：前端体验（用户视角纵深）

让用户真正愿意用这个系统。

- 创建 run 的页面美化（LaunchInput 交互打磨）
- 运行中的进度展示（"正在生成第 3/12 个镜头"）
- 失败/返工的可读展示（"视频生成失败，自动切换 Kling 重试"）
- autopilot 模式的实时状态条（"🔄 自动运行中"）
- 运行完成通知（SSE 已有，前端没做 toast）
- 成片预览播放器
- 移动端适配

**推荐理由：** 后端再强，用户看不到就等于没有。

### 路 C：系统纵深（提能力）

在当前核心链路基础上加能力。

- 多项目并发管理（项目列表、批量操作）
- 用户积分/计费真实对接
- 自定义 provider 配置（用户自己配 API key）
- 成片历史版本管理
- 人工审核工作流（review 时真人可介入修改）
- 模板/预设系统（一键复用配置）
- 国际化（英文界面）

**推荐理由：** 产品化必须走。

---

## 三条路的优先级建议

1. **先走路 A 第一步**：Docker Compose。成本最低收益最高，解决环境一致性问题。
2. **然后走路 B**：让用户看到东西。现在是黑盒跑完一个 run，用户不知道里面发生了什么。
3. **最后走路 C**：能力越强意味着复杂度越高，需要前两步的稳固基础。
